[ Tenancy ]
- the highest level of segementation
(including the defualt compartment)
- can serve as the overarching organization (high level centralization) or represent a distinct project or program (low level distribution)

[ Compartments ]
- resources can only belong to (1) compartment
- resources can be "moved" between components
- resources can span regions 
- used for organization and fine grained access-control
- support nested compartments (up to 6 levels) and policy inheritance (poilcy - Azure Management Groups, GCP Folders)
- I can configure quotas to limit consumption across components // UNIQUE

// Example Quote Policy
set <family> quota <quota name> to <count> in <tenancy || compartment>
set vcn quota vcn-count to 4 in compartment Production

unset - reset back to default settings
zero - sets value to 0; none of these resources can be created

[ Policy ] 
- an entity that specifies which groups can access specific resources and how
- best practice is to assign access at the compartment level
- policy can also be placed at the tenant level

  inspect - observe and monitor; no access to data
  read - read data
  use - modify pre-existing resource
  manage - create/delete; full administrative control

  all-resources
  # Aggregate Resource Types

  db-systems

// Example
Allow group 'Production (Domain)/'Network Admin' to manage virtual-network-family in compartment Sandbox

- some resources have an "aggregator type"; this means access can be granted to the entire family of the type (.e.g volume-family) // UNIQUE
  
- all policy statements begin with allow
- by default all access to resources within compartment is explicitly denied
- policy is inherited between the parent compartment and any child components (think Management Groups Azure)

[ Local Peering Gateway - LPG]
- LPGs must be deployed to each VCN (Step 1)
- LPG Peeing request must be established by one of the VCN (Step 2)
- Route Rules must be created in each VCN Route Table (Step 3)
- Security List must be updated to accept traffic from each VCN (Step 4)
- this is intra region peering
- think AWS/GCP VPC Peering, Azure VNET Peering

[ Remote Peering Gateway - RPG ]
- this is outer region peering
- implemented via DRG

// Workflow
Provision (2) DRGs 
  Provision (2) Remote Peering Connections (RPCs) between both DRGs
     Provision a Connection using the RPCs OIDs
        Create Route Rules in VCN's
          Permit Traffic via Security List in VCN's

[ OCI Functions ] -- UNIQUE
- logical grouping of functions and allocation of resources
- combines configuration context and runtime isolation
- must specify subnets that the function will execute in
- there is no default CSP Provided execution context (like AWS)
- it is ideal to group multiple functions in a single application for efficiency and performance -- UNIQUE
- functions are built and stored as container images in a registry -- UNIQUE
- can be invoked using CLI, signed HTTP request, Oracle SDKs, other OCI services
- on first invocation, OCI function pulls the container and executes the function
- subsequent requests to the same function are directed to the existing container
- after a period of inactivity the container is removed
- container instances can scale up to 60 GB RAM for function execution

// Concept 
  Application
    Function 1 -> Container Image
    Function 2 -> Container Image
    
// Accessing Other OCI Resources Using Functions
Add function to Dynamic Group
  Add the necessary policy for them to be able to access the target service 
    Tokens granted to resources using the group are cached for 15 minutes
